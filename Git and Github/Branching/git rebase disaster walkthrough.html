The Golden Rule of Rebasing — Disaster Walkthrough

Why "never rebase shared history" is stated as an absolute rule, not a guideline.

The rule: Never rebase commits that have been pushed to a public/shared branch. Rebasing replaces commits with new ones that have different SHA-1 hashes, and anyone who already has the originals will have their history broken.

1 Shared starting state

You, Bob, and Carol all have main ending in commits X and Y. You notice two small issues:

  • X's commit message has a typo (projct instead of project)
  • Y has a leftover console.log("DEBUG")

2 You rebase locally to fix both

Using reword for the typo and edit for the debug log:

git rebase -i HEAD~2

reword 4a2b6e7 X: setup projct          ← typo in message
edit   9c1d5f8 Y: add database config   ← debug log in code

The code in your project is essentially unchanged (one log line gone), but Git treats X / Y and X′ / Y′ as four separate commits because hashes determine identity, not content.

3 You force-push to the remote

git push --force origin main

The remote main now points at Y′. From the remote's perspective, X and Y simply don't exist anymore:

But Bob and Carol still have X and Y on their machines.

4 Three failure modes hit your team

You →→ origin: force-push X′, Y′
Bob →→ origin: git pull (Bob still has X, Y locally)
    ⚠ Phantom conflicts on every diff line
Bob →→ bob/login: rebase (anchored to orphan Y)
    ⚠ Branch stranded on dead base
Carol →→ origin: git push (rejected)
Carol →→ origin: git push --force
    ⚠ Your X′, Y′ wiped silently

① Bob's phantom conflicts

Bob has X, Y locally. Remote has X′, Y′. When Bob runs git pull, Git tries to merge two timelines of nearly identical code — but it can't see that X and X′ are "the same." It produces conflicts on every line that differs even slightly. Bob spends an hour resolving fake conflicts.

② Bob's stranded feature branch

Bob's bob/login was anchored to Y, which is now an orphan. When he tries to rebase onto the new main, Git treats his work as descending from Y and applies it onto Y′ — with subtly different bytes. More conflicts.

③ Carol's silent rollback

Carol's push is rejected because her tip predates X′. Not knowing about your rebase, she does the same thing you did: git push --force. This erases X′ and Y′ from the remote entirely. Your fix is gone. Nobody notices for hours.

The signature in the log

After Bob resolves his merge, his git log --graph shows both X and X′, both Y and Y′ — duplicates with different hashes representing the same conceptual changes:

* a8c9def  Merge remote-tracking branch 'origin/main'
|\
| * e7a4f8b  Y′: add database config       ← from your rebase
| * d3c2b1a  X′: setup project              ← from your rebase
* | 9c1d5f8  Y: add database config         ← Bob's original
* | 4a2b6e7  X: setup projct                ← Bob's original (typo!)
|/
* c3c3c3c  C: initial setup complete

The repo now has parallel ghost timelines of the same work. "When did we add the database config?" returns two answers.

Why the rule is absolute, not "use caution"

The trigger here was a typo and a debug log. Tiny. Look at the cost/benefit asymmetry:

Cost / BenefitMagnitude
Benefit of rebasing shared historyMarginal (slightly cleaner messages)
Cost when it goes wrongHours of teammate confusion + potential lost work
Failure visibility on your machineZero — push and everything looks fine
Failure visibility on others' machinesHours later, scattered across the team

That asymmetry — invisible to you, expensive to them — is why this is stated as an absolute rule, not a guideline.

The mechanism in one sentence

Git uses hashes (not content) to identify commits. Once a commit is shared, its hash is its identity — change the identity, and every reference to it elsewhere breaks.

There's no way for Git to know "X and X′ are basically the same thing." To Git, they're four different commits, two parallel histories, and someone has to pick which timeline wins. The merge or rebase machinery treats X and X′ as separate work and demands reconciliation. Rebasing shared history forces that reconciliation onto everyone who already had the originals.
